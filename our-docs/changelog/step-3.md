# Step 3 — Metadata enrichment (done)

Built the metadata-enrichment pass specified in `03_metadata_enrichment.md`.
Step 1's structural nodes now carry the filterable signals retrieval needs:
publication / effective / repeal dates, authority + rank, language,
supersession, and a composite `usable` flag. No LLMs in this stage.

## Pipeline (single runner)

Architectural decision: **metadata is document-level, so extract once per
root and propagate to all children.** With 1.97M nodes spread across only
63k root documents, this turns "1.97M HTML reads" into "63k HTML reads".

1. **Pass 1 — root scan.** Stream `output/nodes.jsonl`, collect every
   `LAW/GUIDE/CASE/TREATY` node. Build `law_id → (source_file, subcorpus, title)`.
2. **Pass 1b — HTML extraction.** Open each root HTML file once, dispatch
   to the matching per-subcorpus extractor (multiprocess fan-out, 8
   workers on macOS fork). Build `law_id → RootMetadata` in memory
   (~63k entries, well under a GB).
3. **Pass 1c — amendment chain walk.** Read `output/edges.jsonl`, group
   `amends`/`repeals` edges by target root, pick the latest-published
   amender per chain → `law_id → superseded_by`. Self-references and
   non-statute sources (KHO, Vero) are filtered out at this boundary.
4. **Pass 2 — rewrite.** Stream every node, attach root metadata +
   authority + composite `usable`, write to `output/nodes_enriched.jsonl`.

## Decisions and tradeoffs

### Document-level vs per-node extraction
The spec is written at the per-node level ("each node has `publication_date`…"),
but every Step-3 field is actually a property of the root document.
Extracting per-root and propagating to children cuts the HTML read budget
by ~30× without any loss of fidelity. The on-disk shape still has the
fields on every node, so downstream consumers see no change.

### Last-resort year fallback
First pass on the full corpus came in at 94.2% `publication_date`
coverage — just under the spec's 95% target. Investigation showed two
patterns: Finlex appendix/annex files with no structured date markers,
and Vero kannanotot whose body says "verovuodesta NNNN" without a
formal "Antopäivä". Added a coarse last-resort scrape (latest 4-digit
year in body, anchored to Jan 1) used only when no structured extractor
matched. Lifted coverage to 99.0% finlex / 99.8% vero. Encoded as Jan-1
so it's distinguishable from precise dates in downstream filtering.

### Statute-source guard on amendment chains
Step 2's regex extractor occasionally labels a KHO precedent as
`amends`/`repeals` a statute. Per the edge taxonomy in
`02_edge_extraction.md`, those edge types only flow `finlex/* →
finlex/*` (and in practice only from `laki`/`asetus` subcorpora). The
chain walker now drops edges whose source root sits outside a statute
subcorpus — defensive validation at the Step-3 boundary, not a fix to
Step 2's regex. Without it, two `Tuloverolaki` and `Työsopimuslaki`
versions got bogus KHO-case amenders.

### `superseded_by` resolution granularity
`superseded_by` points at a **root law_id**, not a specific section. The
spec example ("amended statute shows superseded_by pointing to the
amender") matches this. Per-section supersession needs version-aware
section IDs that Step 1 doesn't emit; deferring that until the
graph-store schema decides whether to track it.

### Vero "tämä ohje korvaa" markers
The Vero extractor exposes `has_korvaa_marker(html)` for future use but
doesn't yet resolve the referenced document — that requires either an
LLM pass or a fuzzy title-match against the corpus index. Not blocking
for v1: Vero supersession will fall out of Step 2's `interprets` chain
once that's exercised in retrieval.

### Re-runnability
The runner is idempotent. If `output/edges.jsonl` is missing (Step 2
not yet done), it logs a warning and skips supersession. The first
Step-3 run for this corpus ran *before* Step 2 finished — re-running
after Step 2 just rewrote `nodes_enriched.jsonl` with `superseded_by`
populated, no other changes needed.

## How to run

```bash
# Full enrichment
.venv/bin/python -m scripts.enrich_metadata --workers 8

# Smoke run (process only N root documents)
.venv/bin/python -m scripts.enrich_metadata --limit-roots 50

# Validate output
.venv/bin/python -m scripts.qa_check_metadata
```

## Sequencing notes

This step makes the retrieval reranker's filterable signals real.
Downstream usage:

- **Step 4 (graph loader)** reads `nodes_enriched.jsonl` directly into
  the graph store and copies the authority/date columns onto the
  `VectorRecord` payload (schema already mirrors these fields in
  `src/models.py`).
- **Step 5 (annotated context)** uses `authority_rank` for source
  ordering and `publication_date` for "newer trumps older" annotations.
- **Step 7 (degree caps)** is unaffected; `metadata.degree` is filled by
  Step 4, not Step 3.
- **Step 8 Verifier (L7 conflict surfacing)** needs `authority_rank` to
  flag conflicts between high- and low-tier sources — that signal is
  now live.

Re-run Step 3 whenever Step 2 produces a new edge set (re-resolves
`superseded_by`) or when a new corpus snapshot lands (re-extracts root
metadata).
